1. 커밋 메시지에 대해 강제력을 가지고 있는 툴을 사용하고 있습니까?
-> 깃 툴의 pre-commit 도구를 통해 통일된 커밋 메시지에 대한 관리를 하는 것을 추천합니다.

2. 브랜치 명을 제 3자, 혹은 팀원들이 좀 더 알기 쉽게 구성하는 것이 좋을 것 같습니다. 
-> 지금 브랜치 이름을 보면, 브랜치만 보고 기능을 명확히 파악할 수 없습니다. 같이 일을 하다보면 어떤 브랜치에 어떤 기능이 있는지 알 수 있어야 하는 것이 좋습니다. 현업에서는 이슈 번호, 브랜치 이름에 기능에 대한 명칭 넣습니다. 앞으로는 어떤 브랜치에 어떤 기능이 있는 지를 알 수 있게 구성하는 것이 좋을 것 같습니다.

3. 컨트롤러의 url을 나누는 방법을 통일성 있게 구성하면 좋을 것 같습니다. 
-> 현재 Api를 보면 restApi를 만드는 방법이 아쉽습니다. 통일성 있는 
Api명은 RequestMapping을 통해 통일성 있게 구성하는 것이 좋으며, 하위로 
restController Mapping을 사용을 사용하며 url의 확장성을 고려하는 것을 추천합니다.

4. Util 패키지 안에 왜 Dto 클래스 파일이 있는 지 궁금합니다.
-> Util 패키지 안에 작성된 여러  파일이 진짜 쓰이고 있는지 다시 확인해보는 것이 좋을 것 같습니다. util 안의 파일들은 쓰임에 맞게 다시 정리가 필요해보이며, 
url의 필터 인터셉터, aop에 대해 알아보시면 좋을 것 같습니다.

5. Security Configuration Bean을 보며 Gradle을 확인했는데, 시큐리티 버전 명시가 명시되어 있지 않습니다.
-> 버전에 따라 SecurityConfigurationAdapter를 활용할 수 없는 경우가 있습니다. Security 버전을 확인하고 에 맞는 기능을 확인 하시는게 좋을 것 같습니다. 해당 부분의 질문이 많이 나오고 있으니 꼭 숙지하시기 바랍니다.
